Factory milestone 2 / Canary event to BB triage

Wire the first live Canary signal into a Bitterblossom Sprite triage run.

The target outcome is a low-risk Canary event that produces a durable BB run, executes the report-only canary-triage task on Sprites, and leaves a readable REPORT.json. Canary stays the observability substrate; Bitterblossom owns execution.

Chosen Design

Canary eventUse existing signed webhook delivery for a bounded event payload.
BB ingressVerify HMAC, filter to reportable Canary events, dedupe by delivery id or subject.
Run ledgerCreate one canary-triage run before ack.
Sprite agentAPI-auth report-only triager queries Canary and inspects repo context.
ReceiptStore REPORT.json, visible through BB run/artifact surfaces.

Stop Conditions

  • No secret values in logs, argv, commits, or reports.
  • No code edits, branch push, deploy, issue creation, or incident resolution by the triage agent.
  • Stop if BB requires workload-specific Rust instead of task config/card files.
  • Stop if live run cannot produce a report and classify the exact blocker.

Acceptance

  • BB has a canary-triage task, API-auth agent binding, manual trigger, webhook trigger, budget, and required artifact contract.
  • Fixture webhook creates one run and redelivery creates no duplicate run.
  • Task card requires REPORT.json with Canary subject, delivery id, service, repo mapping, evidence, hypotheses, and residual uncertainty.
  • A live Canary-triggered event reaches BB and the resulting run can be inspected through bb runs show and bb artifacts read.

Adversarial Review Focus

  1. Can a filtered or duplicate event create extra paid runs?
  2. Can a reflex agent inherit mutation authority through credentials, card wording, or recommended commands?
  3. Does the report link back to durable Canary state rather than trusting the webhook payload?
  4. Can an operator inspect the result without SSH or log spelunking?
  5. Does the live path depend on untracked local state that production Fly cannot see?
