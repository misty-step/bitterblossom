Plane Excision: Runtime Volume Contract

Goal: stop baking Misty Step production plane files into the product image. First slice changes Docker/Fly to expect a runtime plane directory at /app/plane, documents the one-time Fly volume migration, and adds a gate that fails if Dockerfile ever reintroduces COPY plane.

Why This Slice

The current publicability blocker has two layers: the image embeds production instance config, and the repo still tracks that instance config. Removing the image embed is the smaller deployment-safe cut: it makes budget and task-card changes runtime data, not image rebuilds. Deleting tracked plane/ follows after the gate and local workflows no longer require it.

Change Shape

SurfaceBeforeAfter
Docker image COPY plane ./plane No production plane files in image; BB_PLANE_DIR=/app/plane.
Fly mount Volume mounted at /app/plane/.bb for ledger only. Volume mounted at /app/plane for config plus state.
Gate No publicability check. verify.sh rejects COPY plane.
Docs Deployment docs say checked-in plane is production config. Deployment docs require operator-supplied instance plane and migration.

Acceptance

  • rg "COPY plane" Dockerfile returns no match.
  • fly.toml mounts bb_plane_data at /app/plane.
  • ./scripts/verify.sh passes and includes the publicability guard.
  • Docs explain how to pre-stage config and state on the Fly volume before deploy.

Stop Conditions

Do not deploy in this slice. The Fly volume migration is outward-facing and must be run deliberately by an operator after the PR lands.
