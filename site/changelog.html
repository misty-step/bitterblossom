About the version history

A real discontinuity: the v3 Rust rewrite has no tag yet

Bitterblossom has tagged GitHub releases going back through v1.79.0 and earlier — but every one of them belongs to the retired Elixir/Python conductor, not the current Rust bb binary. The rewrite has been running as the plane's actual mechanism for months without a version number: Cargo.toml still carries the 0.1.0 placeholder from cargo init.

docs/release-policy.md (bitterblossom-097, merged 2026-07-04) fixes the mechanism problem: a semver policy scoped to what an operator or agent actually depends on (CLI flags, config fields, /api/* routes, the ledger schema) and scripts/bb-cut-release, a script that computes the next version and — only with --live, only off a clean tree, never silently — tags, pushes, and cuts a real GitHub Release. What it deliberately does not do is pick the first v3 version number; that's an operator decision, left open on purpose rather than defaulted.

Evidence: docs/release-policy.md, scripts/bb-cut-release, and Cargo.toml's current 0.1.0.

2026-07-04 - release mechanism shipped

A real, repeatable way to cut the first v3 release

  • Shipped scripts/bb-cut-release and docs/release-policy.md (bitterblossom-097, first slice): dry-run by default, refuses to run live on a dirty tree, off master, or against an existing tag.
  • Publishing a release already triggers Landmark synthesis (.github/workflows/landmark-release.yml) into docs/releases/ — this slice supplies the missing piece: an actual way to cut the release that fires it.
