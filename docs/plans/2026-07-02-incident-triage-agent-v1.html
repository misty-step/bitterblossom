PLAN / INCIDENT TRIAGE AGENT V1

operator-decided design, 2026-07-02

LANE triage-build

Ship one dedicated Canary incident responder on the production BB plane.

The runtime task wakes on Canary incident webhooks, uses GLM 5.2 via OpenRouter, writes progress back to Canary, fixes allowed Misty Step repos through PR, asks Cerberus for advisory review, waits for CI, merges, and records the honest V1 stop point.

CHOSEN DESIGN

Command-wrapper task plus model lane prompt. BB owns the iteration guard, environment validation, and artifacts; GLM owns incident reasoning and code work.

QUALITY SYSTEM

Live oracle: synthetic low-severity Canary incident triggers a production BB run through hypotheses and Canary writeback.

Structural gate: repo contract tests plus ./scripts/verify.sh.

Critic: fresh artifact-only review if the diff touches runtime code beyond config/card/wrapper.

STANDARDS

Mechanism stays in BB; workload judgment stays in card/wrapper. Secrets travel through declared env only. No broad repo scope.

ACCEPTANCE

Production plane has task, agent, card, budget, and wrapper. Whitelist is canary, bastion, powder only.

VERIFY

Run check locally, validate production config, trigger drill, capture BB run id and Canary timeline/writeback evidence.

COMMUNICATE

Report shipped surfaces, live evidence, V1 scope limits, and friction tickets.

STOP IF

Production auth is actually unavailable, the model cannot write back, task config fails check, or the incident target is outside the whitelist.

CURRENT STATE

canary-triage exists as report-only. Review agent uses command wrapper patterns with explicit GH token forwarding.

CHANGE SHAPE

Add incident-triage agent/task/card and wrapper, contract tests, public fixture copy, and production volume config.

PROOF SURFACE

bb check, wrapper tests, production run receipt, Canary timeline annotation/claim events, and final git status.

SCOPE

Goal: full flow through merge for established auto-merge/CI repos; deploy and prod QA only where auto-deploy on merge is already real.

Non-goals: new paging, broad repo rollout, new Canary APIs, or generic runtime service lookup.

Invariant: maximum three fix attempts per incident, then write escalation-needed and notify BB.

ANCHORS

scripts/cerberus-review-wrapper.sh for command-wrapper token handling.

docs/operations/README.md for production volume update path.

docs/agent-inspection-cli.md in Canary for claim and annotation writeback.

EXECUTION PLAN

1 frame

Read live task, wrapper, Canary writeback, and production ops contracts. Confirm model id and production access.

Evidence: docs and live OpenRouter model catalog.

2 build

Add task/card/agent/wrapper and tests. Keep BB Rust changes limited to contract support only if needed.

Evidence: diff, focused tests, full gate.

3 prod

Copy config to /app/plane, run remote bb check / recover or restart path as required.

Evidence: remote command output without secrets.

4 drill

Use Canary drill machinery to emit a low-severity incident and watch BB run through hypotheses plus writeback.

Evidence: BB run id, timeline entries, and residual V1 limits.
