PR REFLEX STORMS / BITTERBLOSSOM

A PR open, ready-for-review, or update should automatically create a submission and enqueue the full adversarial storm.

Chosen design: extend webhook triggers with a generic submission_storm action. The trigger remains data-owned in task.toml; the Rust spine only maps event payload fields to the existing submission and canonical storm run mechanics.

CHOSEN DESIGN

Add trigger-time expansion, not a review conductor: authenticated PR webhook -> one submission -> required verdict runs keyed as storm:<submission>:<kind>.

QUALITY SYSTEM

Proof: synthetic webhook oracle, checked-in plane contract, full repo gate, and a fresh diff critic before merge.

Premise source: sha256:4eb48ff00fbb8d37f20176d5afc9ed8d25e756904a19755edba20f102ee429db docs/plans/2026-06-11-039-submission-loop.md

External source: GitHub pull_request webhook docs, checked 2026-06-16.

STANDARDS

Deep module, small surface. No workload-specific branches in dispatch. Reflex agents stay API-auth and hermetic. Large PRs are reviewed, not silently filtered.

ACCEPTANCE

A signed in-scope PR webhook creates one control run, one submission, and one pending run per required gate member. Redelivery creates no duplicate submission or storm members.

VERIFY

cargo test --test ingress, cargo test --test lifecycle_reflex, bb check plane, and ./scripts/verify.sh.

COMMUNICATE

Report exact trigger config, synthetic delivery result, gate/member state, and residual production deployment risk.

STOP IF

The design needs a semantic orchestrator, needs GitHub API writes, weakens auth policy, or cannot stay inside the existing submission/gate data model.

CURRENT STATE

review has a PR webhook but creates one review-coordinator run. Storm members and bb gate are manual. The checked-in review trigger filters /pull_request/additions <= 4000.

CHANGE SHAPE

Add a trigger action type that derives change, rev, repo, and freshness version, then enqueues required verdict tasks. Update the checked-in PR trigger to use it and remove the additions cap.

PROOF SURFACE

A synthetic oversized PR payload must be accepted and must materialize the submission plus canonical storm run rows without running external agents.

SCOPE

Goal: PR lifecycle events automatically start adversarial review swarms.

Non-goals: automatic fixes, automatic merge, review-comment ingestion, GitHub status posting, model selection changes, or per-repo gate profiles.

Invariants: HMAC required, filters fail closed, duplicate deliveries are idempotent, gate policy remains mechanical, and large PRs are budgeted instead of skipped.
