## Findings

### Finding 1: `sprite exec` passes bash flags as sprite flags (FIXED)

- Severity: P1
- Status: Fixed during audit
- Observed: `Conductor.Sprite.exec/3` called `sprite -o O -s S exec bash -lc CMD`. The sprite CLI parsed `-lc` as its own flag and errored: "unknown shorthand flag: 'l' in -lc".
- Fix: Added `--` separator before `bash` in sprite.ex line 30.
- Evidence: run-698-1773687901 failed in 1 second with workspace_preparation_failed
- Tests: 359 pass after fix

### Finding 2: Codex non-zero exit kills run despite valid PR

- Severity: P1
- Existing issue: Related to #675 (builder artifact protocol), #685 (builder completes without artifact)
- Observed: Codex exited with code 1 after producing 4 commits and opening PR #699. The conductor treated this as `builder_dispatch_failed` and never scanned for the PR. CI is green.
- Expected: The conductor should detect and adopt the PR even when the builder harness exits non-zero. The PR is the artifact that matters.
- Why it matters: Every Codex builder run that exits non-zero — for any reason (test failure, lint, timeout) — is treated as a total loss. The work is thrown away and the workspace cleaned. This is the biggest reliability gap in the factory.

### Finding 3: Conductor never detects PR during building phase

- Severity: P2
- Existing issue: New
- Observed: PR #699 was open for ~13 minutes before the builder exited. The conductor's `pr_number` remained null the entire time. PR detection only happens after builder completion.
- Expected: The conductor should poll for PR creation during building so it can track progress.
- Why it matters: If the builder fails, the conductor has no record of the PR. The run record says "failed, no PR" when a PR exists. Operator confusion.

### Finding 4: `run-once` is deprecated but still works

- Severity: P3
- Observed: "warning: run-once is deprecated. Prefer `mix conductor start` with fleet.toml."
- Impact: The factory-audit skill (just updated) and docs still reference `run-once`. Should update docs or keep `run-once` supported.

### Finding 5: Factory-audit skill and snapshot script were stale

- Severity: P2
- Status: Fixed during audit
- Observed: SKILL.md referenced `scripts/conductor.py` (doesn't exist), required `autopilot` label (gate removed). `collect_run_snapshot.py` called `scripts/conductor.py show-runs` instead of `mix conductor show-runs`.

## Reflection

- What Bitterblossom did well:
  - Workspace preparation was fast (1s after the `--` fix)
  - Builder produced high-quality output: 835 LOC feature + 499 LOC tests + docs + Makefile
  - CI passed on first push (Go, Elixir, Shell, YAML, hooks, trufflehog)
  - Heartbeat was reliable throughout the 25-minute build

- What felt brittle:
  - The Codex exit-code-as-success-signal design. The builder did excellent work and the conductor declared it a failure.
  - No PR detection during building means the conductor operates blind for the longest phase.
  - The `--` separator bug was a one-character fix that blocked all dispatch.

- What should be simpler next time:
  - PR-first success detection: if a PR exists on the run's branch, the run is not a total failure regardless of exit code.
  - The factory-audit skill should be tested as part of conductor CI, not discovered stale during audits.
