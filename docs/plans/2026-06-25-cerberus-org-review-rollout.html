PLAN / CERBERUS REVIEW ROLLOUT

2026-06-25 · shape before deliver

Target outcome

Bitterblossom can run Cerberus as the selected PR review workload across Misty Step and Phrazzld repos without opening an uncontrolled trigger or spend surface.

Keep the product boundary clean: Cerberus is the current external specialist review runner, while Bitterblossom owns webhook filtering, dedupe, budgeting, dispatch, and receipts. This slice makes the checked-in review task a Cerberus command wrapper and adds generic ingress denial filters for bots/noisy events.

Chosen Design

Use the existing command harness to invoke cerberus review-pr from the review workload. Remove the old submission-storm fan-out for this trigger, then add generic not_any_of webhook filters and configure org, bot, draft, action, file, and size controls in plane/tasks/review/task.toml.

Quality System

Standards: no workload judgment in Rust; no duplicated review brain in Bitterblossom; org rollout is controlled by task config; secrets stay in env/stdin plumbing.

Proof: fixture webhook tests, command-wrapper smoke, bb check, ./scripts/verify.sh, and one artifact-only fresh critic.

Current State

plane/tasks/review currently uses a Kimi/Pi card that reviews and comments directly via gh. It is allowlisted to one repo and has no deny-list predicate for Dependabot-like senders.

Change Shape

Add one generic filter operator; replace the review task's agent/card with a Cerberus command wrapper; widen config to org owner allowlist plus bot/size/file gates.

Proof Surface

tests/ingress.rs proves filtered deliveries do not create rows. A local dev-plane Cerberus fixture run proves the wrapper surfaces artifacts without GitHub side effects.

Scope

Goal: make the repo-ready review workload Cerberus-backed and safe to widen from one repo to the two orgs in controlled stages.

Non-goals: live org webhook installation, Cerberus product changes, per-repo policy engine, auto-merge, or posting to every open PR today.

Invariants: one run per PR head SHA, webhook filters acknowledge out-of-scope events, reflex agents use API auth, and the plane does not encode review judgment.

Risks + Rollout

Spend: bot bursts or large PRs create queued work. Mitigate with sender deny, action/draft/size/file filters, one-run-per-head, max runs/day, and task parking.

Posting noise: keep fixture/dry-run proof before production posting; Cerberus handles idempotent GitHub projection.

Host drift: wrapper fails if Cerberus or its harness is missing. Preflight should expose command-harness binary availability; production host readiness remains a deployment check.

Verification System

Claim: BB admits only controlled org PR review events and dispatches Cerberus as the review runner.

Falsifier: Dependabot or out-of-org delivery creates a run; one PR head creates more than one review run; task still points at the native reviewer; wrapper posts during fixture/dry-run smoke.

Driver: focused Rust tests, bb --config plane check, optional local fixture bb run review, and ./scripts/verify.sh.

Grader: no run rows for filtered events, one row for valid org event, Cerberus artifacts in REPORT.json, and green gate.

Evidence packet: command transcripts and final gate output in closeout.

Cadence: before merge and again before any production webhook widening.

Fresh Critic Prompt
