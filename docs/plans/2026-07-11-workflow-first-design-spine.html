Product direction · 2026-07-11 · Powder: bitterblossom-workflow-ux-lock

Bitterblossom becomes the workflow roster for unattended agent operations.

The work is to replace a task-and-run dashboard with a workflow-first control plane whose configured agents, triggers, live instances, authority, spend, evidence, and history are understandable at a glance.

Work contract

Target outcomeCanonical product truth, an executable Powder program, and a full design catalog awaiting operator verdict.
Chosen product shapeWorkflow is primary. A trigger commissions an agent with a natural-language goal; its result routes to another agent or completion.
Proof surfaceRepo diff and gate, Powder readback, and a rendered 12–20 option lab with the shipped baseline at desktop and phone.
Stop conditionStop before convergence or implementation. The operator chooses, kills, and mutates catalog options.

Direction locks

  • Initially, Bitterblossom is strictly for unattended trigger-driven workflows. Supervised local-agent registration is deferred.
  • The configured ontology is Workflow, Trigger, Step, Route, and reusable Roster Agent. Runtime truth is Run and Result.
  • Goals are natural-language programs enhanced agentically with acceptance evidence, boundaries, and assumptions.
  • Live configuration is revisioned database state. Declarative files are import/export and optional GitOps surfaces.
  • Roster supplies pinned agent definitions. Mint is the primary credential broker. Workflow grants narrow the Roster capability ceiling.
  • PR Review is the golden workflow. Canary incident resolution follows as the first full remediation loop.

Design fence

ArtifactThe authenticated Bitterblossom operator application: workflow roster, workflow detail, live/history runs, agents, spend, and goal-first creation.
FixedTruthful product nouns; light and dark; WCAG AA; keyboard reach; static zero-build prototype; real PR Review and Canary content; no Misty Step/Aesthetic religious constraint.
VaryingInformation architecture within the locked navigation, graph grammar, hierarchy, density treatment, typography, shape, color architecture, and interaction model.
DialsVariance: medium-high. Motion: low and state-tied. Density: high but layered.
Distinctive requirementThe interface must make configured workflow topology and live agent composition legible without turning into either a generic KPI dashboard or a node-editor toy.

Design bench

Six blind philosophies each propose three reusable system directions. After structural deduplication, the catalog retains 12–20 candidates plus the shipped baseline. Every option renders the same complete product gallery and supports light and dark themes.

  1. Distinctive product-native frontend
  2. Metric-driven anti-default dashboard
  3. Flat editorial/minimal operator system
  4. Industrial brutalist control plane
  5. Macrostructure-first anti-slop system
  6. Impeccable discipline-led product interface

Acceptance and evidence

  • VISION and supporting docs contain no active contradiction about supervised scope, files as authority, or task-first UX.
  • Powder has a governing epic, executable child cards, relations, and explicit dispositions for superseded work.
  • The design catalog includes the shipped state separately and at least twelve structurally distinct candidates.
  • Each candidate exposes workflows, one workflow graph, live/history runs, agents, spend, and goal-first creation using truthful content.
  • Desktop and phone render without overlap or console error; theme and navigation controls work.

Risks under review

  • The existing noir-ledger contract may preserve density at the cost of comprehension. It is baseline evidence, not a binding visual religion.
  • A visual workflow graph can become decorative spaghetti. Candidates must keep the stable configured graph distinct from selected-run overlays.
  • Agentic creation can hide authority. Every proposal must keep agent revisions, grants, budgets, and activation review inspectable.
  • The current runtime remains file-defined while the target becomes database-authoritative. Documentation must mark current truth versus direction without pretending migration already shipped.
