# Context Packet: The submission loop — verdict storm, mechanical gate, bounded rounds

## Goal

Completed agent work gets autonomously quality-assured and landed: a
storm of independent verdict-producing tasks attacks a submitted change,
a mechanical gate decides blocking vs. clear, the implementing agent
loops on the report, and the loop provably terminates — no human reads
the code, no pull request is required for coordination.

## Non-Goals

- **Push-event reflex ingress** (GitHub App or post-receive hook on a
  self-hosted remote). v1 is dispatch-driven by operator decision
  (AskUserQuestion, 2026-06-11): the implementing agent fires the storm.
  The trigger seam is the plane's existing trigger table; reflex arrives
  as a follow-up ticket once the plane has a durable home.
- **Plane deployment.** The loop runs from a laptop plane + sprite fleet.
- **PR integration.** No comments posted anywhere. A PR may exist for
  external visibility; it is never the channel. (The fallback "old mode"
  — PR + CI + comment storm — is explicitly not built here.)
- **git-notes export** of verdicts. The ledger is canonical; a notes
  export step can ride in a marshal card later without spine changes.
- **An agentic-QA storm member** (drives the running app). Needs a
  per-repo runnable-app harness; v2.
- **Multi-repo gate config.** One `[gate]` per plane in v1.

## Constraints (invariants that survive)

- **No workload logic in the spine.** Verdict storage, gate arithmetic,
  and round bookkeeping are generic data mechanics (like budgets).
  What a reviewer looks for, what counts as evidence, how findings are
  phrased — cards only.
- **Model & auth policy holds**: storm members are reflex-class work —
  api-auth agents, cheap OpenRouter models, hermetic. (They run via
  `bb run` in v1, but they are built as reflex-eligible tasks so the
  later trigger swap is config-only.)
- **Spine LOC budget ≤ 5k** (currently 4,032). If this change cannot fit
  in ~+600 LOC, stop and surface the budget conversation explicitly.
- **Unparseable output is failure** — a verdict task whose result is not
  valid verdict JSON fails the run; never a silent pass.
- **`change` keys are opaque strings** to the spine. v1 drivers pass a
  stable topic key (branch name or ticket id) plus the per-round `rev`
  SHA; the jj migration later swaps in change IDs with zero spine change.

## Repo Anchors

- `src/spec.rs` — config loading + load-time validation; where
  `verdict = "<kind>"` on TaskSpec and `[gate]` on PlaneSpec land.
- `src/dispatch.rs` — `attempt_on_host` collecting phase; where a
  successful verdict-task run parses result.md into a verdict row.
- `src/ledger.rs` — table + accessor patterns (runs/attempts/budget
  tables; CAS transitions); the `verdicts` table follows these.
- `src/serve.rs` — read API shapes (`/api/runs`, `tasks_view`); add
  `/api/gate` and `/api/verdicts` in the same style.
- `src/main.rs` — CLI verb patterns (`runs list --json`); add
  `bb verdicts list` / `bb gate`.
- `plane/tasks/review/card.md` + `task.toml` — the exemplar verdict task
  (its JSON summary final answer is already verdict-shaped); storm cards
  derive from it, minus the `gh pr comment` step.
- `tests/e2e_local.rs` — stub-harness e2e pattern for loop tests.
- `CLAUDE.md` (Verification) — the live-evidence contract this work
  extends.

## Alternatives

1. **Old mode: PR + GitHub CI + comment storm.** Open a PR, wait for
   checks, bots comment, implementer reacts. Works today, zero spine
   change. Fails the premise three ways: vendor-locked coordination,
   markdown threads as the data model (lossy for agents), and no
   mechanical termination — the needling loop lives in prose. Kept as
   the documented fallback, not built.
2. **Orchestrated fan-out/join in the spine** (run groups: one event
   spawns N runs, a join collects them). Fails the no-judgment
   constitution — joins, round state machines, and report assembly are
   workflow semantics that grow without bound (the Python conductor died
   of exactly this). Rejected.
3. **Pure git-notes blackboard, no spine change** — storm cards push
   verdict notes; the gate is a script reading notes. Most git-native,
   zero LOC. Fails operationally: notes refs contend under parallel
   writers, sprites need push credentials to write notes, nothing is
   queryable from the API/ledger, and the termination rules end up in
   card prose — policy as intent, the exact failure this arc exists to
   kill. Rejected for v1; notes become an *export* later.
4. **Blackboard via ledger: verdicts as data, gate as query, loop driven
   by the implementing agent** (chosen). Each storm member is an
   ordinary task whose final answer is verdict JSON; the plane records
   it mechanically; `bb gate` evaluates required-set + termination rules
   as pure data policy; the driver (implementing agent) fires rounds and
   acts on the report. Coordination through the artifact, no
   orchestrator. The spine grows one table and one evaluator — both
   generic.

## Design

**Verdict tasks.** `task.toml` gains `verdict = "<kind>"` (e.g.
`correctness`). For such tasks, a successful run's parsed result MUST be
JSON:

```json
{"verdict": "pass|blocking|advisory",
 "findings": [{"severity": "blocking|serious|minor",
               "file": "src/x.rs", "line": 42,
               "claim": "...", "evidence": "...",
               "regression": false,
               "fingerprint": "<carried from prior report, else omitted>"}]}
```

Dispatch (collecting phase) parses and records a `verdicts` row:
`(id, run_id, task, kind, change_key, rev, round, verdict, findings_json,
created_at)`. `change_key`, `rev`, and `round` come from the trigger
payload (contract: `{"repo": "...", "change": "<stable key>",
"rev": "<sha>", "round": N, "base": "<sha>", "context": "..."}`); the
spine treats all three as opaque data. Missing payload fields or
unparseable verdict JSON fail the run with the raw output preserved.
Fingerprints: plane computes `sha256(kind|file|claim)` when absent;
cards are instructed to copy fingerprints verbatim from the prior report
when re-raising.

**The gate.** `plane.toml` gains:

```toml
[gate]
required = ["verify", "correctness", "security", "simplification", "product"]
max_rounds = 3
```

`bb gate --change <key> [--json]` (and `GET /api/gate?change=`)
evaluates, for the highest round with any verdicts:

- any required kind missing at that round → `pending` (names the gaps);
- round 1: any `blocking` finding → `blocked`;
- round N>1: a finding is blocking-eligible only if its fingerprint was
  blocking in round N−1 (unfixed) or it carries `"regression": true`.
  Fresh non-regression blockers are **demoted** to advisory and labeled
  as demoted in the output — this is the monotone rule that kills
  oscillating nitpicks mechanically;
- round > `max_rounds` with eligible blockers → `escalate` (fires the
  notify webhook once);
- otherwise → `clear`.

Output is the report: findings grouped `blocking / demoted / advisory`,
with rounds, costs (joined from runs), and what's missing. The
fix-before-merge / log-for-later / not-worth-recording triage maps to
blocking / advisory-the-driver-files / demoted-and-dropped.

**Storm config (pure files).** Five tasks in `plane/tasks/`: `verify`
(card: run `./scripts/verify.sh` in the workspace, emit verdict from the
result — an LLM-wrapped shell command in v1; a no-LLM `command` harness
is a noted future economy), `correctness`, `security`, `simplification`
(derived from the existing review card, single pass each, different
model families across them for decorrelation — Kimi K2.6 / DeepSeek V4 /
GLM per current bindings research), and `product` (does the diff serve
the stated goal in `context`; scope creep; missing acceptance evidence).
Each binds its own sprite host (`bb-polisher`, `bb-polisher-2`, …) so
storm members parallelize — host leases serialize per host, so
parallelism is a hosting choice, not a spine change. All are
`[[trigger]] kind = "manual"` in v1 with reflex triggers addable later.

**The driver (convention, documented in docs/spine.md).** The
implementing agent, on judging work complete: pushes its branch; for
round 1..=3: fires the five storm members as parallel `bb run`
background processes with the payload contract above; runs `bb gate`;
on `clear` → squash-land and push; on `blocked` → fix, push (new `rev`,
same `change`), next round, carrying the report (with fingerprints and
dispositions) into each card's `context`; on `escalate` → stop, the
operator is notified. Advisory findings get filed to `backlog.d/` by the
driver before landing. The driver is prose + a few commands — judgment
stays with the agent, arithmetic stays in `bb gate`.

**Why this decomposition**: CI, review, red-team, and product critique
are all the same shape (verdict task); the merge gate is a query;
"waiting for CI before the storm" collapses into wave ordering the
driver controls (fire `verify` first if token thrift matters; v1 fires
all five — a red `verify` makes the gate `blocked` regardless).

## Oracle (Definition of Done)

Commands that must all exit 0:

- `./scripts/verify.sh` — the repo gate, including new tests:
  - verdict parsing: valid JSON → row recorded with payload's
    change/rev/round; invalid JSON on a verdict task → run failure with
    raw output preserved;
  - gate rules, each as its own test: `pending` on missing kind;
    round-1 `blocked` on any blocking finding; round-2 demotion of a
    fresh non-regression blocker; round-2 `blocked` on a carried
    fingerprint; `regression: true` blocks in any round; `escalate`
    past `max_rounds`; `clear` when required set present and no
    eligible blockers.
- `bb check` on `plane/` validates the five storm tasks and `[gate]`.
- LOC tripwire: `find src -name '*.rs' | xargs cat | grep -vc '^\s*$'`
  ≤ 5000.

Live evidence (the verification harness is `scripts/verify.sh` plus the
CLAUDE.md live-evidence contract; both updated as part of this work):

- **Full loop on a seeded change in this repo**: a branch with ≥2

## Risks + Rollout

- **Fingerprint fragility.** Hash(kind|file|claim) misses re-phrasings;
  carried fingerprints depend on card discipline. Mitigation: cards
  receive the prior report verbatim and are graded (termination drill)
  on copying fingerprints; accept some false-fresh findings — they get
  demoted, which fails safe (toward landing, with the finding preserved
  as advisory). Watch the first ten real loops for demotion abuse.
- **Demotion failing toward landing is a real bug class**: a genuine
  round-2 discovery gets demoted unless flagged `regression: true`.
  Mitigation: `regression` is the falsifiable escape hatch and the
  quorum is cheap (any reviewer can flag it); residual risk accepted
  consciously — the alternative (fresh blockers always block) is the
  unbounded needling loop the operator explicitly rejected.
- **Spine growth.** Verdicts + gate ≈ +400–600 LOC against a 968-LOC
  headroom. Tripwire in the oracle; if exceeded, stop and renegotiate
  the budget rather than trimming tests.
- **Stop conditions for the builder**: payload contract needs a field
  not listed here → stop and surface; gate rules need workload-specific
  branches (e.g. "security findings never demote") → stop, that's
  config-vs-constitution tension to resolve explicitly, not improvise;
  storm cost per round exceeds ~$1 → stop and rebind models before
  iterating.
- **Rollback**: the table and gate are additive; reverting the squash
  commit restores the prior spine. Storm tasks are files; deleting them
  removes the workload. No migration risk — new SQLite table only.
