Get started

The one-minute, zero-credential golden path: no secrets, no remote substrate, no network. It validates the plane, dispatches a task, records a run, exposes status, reads the run bundle, and ends at bb doctor — onboarding isn't done at build/install, it's done when doctor proves the plane is actually live.

Install

Build from source

Bitterblossom's Rust rewrite hasn't cut a tagged release yet (Cargo.toml is still 0.1.0; the tags you'll see on GitHub back through v1.79.0 belong to the retired Elixir/Python conductor, not this binary — see the release notes for the discontinuity). Until a v3 tag exists, build from a clone: 

git clone https://github.com/misty-step/bitterblossom.git
cd bitterblossom
cargo build

Once a pinned release is cut, the tracked-release path will be: 

cargo install --git https://github.com/misty-step/bitterblossom --tag v<version> --locked

Quick start

Run the zero-credential local plane

examples/local-plane/ is a zero-credential local plane: a command-harness agent whose inline script writes a small REPORT.json. Validate the config, dispatch a task, and read the run back: 

./target/debug/bb --config examples/local-plane check --json
./target/debug/bb --config examples/local-plane preflight hello --json
./target/debug/bb --config examples/local-plane run hello --payload '{"ok":true}' --json
./target/debug/bb --config examples/local-plane status --json
./target/debug/bb --config examples/local-plane runs show <run-id> --json
./target/debug/bb --config examples/local-plane artifacts read <run-id> REPORT.json --json

<run-id> comes from the run --json output above.

Verified live

bb doctor: onboarding ends here

./target/debug/bb --config examples/local-plane doctor --json
./target/debug/bb --config examples/local-plane serve
./target/debug/bb --config examples/local-plane doctor --expect-serve --json

bb doctor composes every check above into one pass/fail verdict: config loads, the ledger is reachable and schema-current, every task's declared secrets and command-harness binaries preflight clean, and — with --expect-serve, run from a second terminal once bb serve for this config is actually running — its unauthenticated /health and / routes answer. Each failing check names concrete remediation; the command exits non-zero if anything fails.

Beyond the local plane

Copyable workload templates

examples/demo-plane/ is a complete commented production-shaped config dispatching to a remote substrate: a Fly Sprite restores checkpoints, syncs repos, and executes the harness over WebSocket exec. 

./target/debug/bb --config examples/demo-plane check
./target/debug/bb --config examples/demo-plane run demo

Credential-free-to-validate templates for real workloads live under examples/: a pull-request review factory (review-factory-plane), a Roster Cerberus role materializer (roster-cerberus-plane), a Canary-style incident responder (canary-responder-plane), and a docs-drift watcher (docs-sync-plane) — each with sample payloads and a local validation recipe. Copy one when the trigger shape and report contract already match your workload; author a custom task when they don't. New workloads should stay config only: plane/task/agent files plus lane cards, never Rust changes.

For agents

The portable operator skill

Agents that need to operate Bitterblossom can load the portable skill folder at skills/bitterblossom/ — copy or symlink the whole folder (not just SKILL.md) into a harness skill root; the recipes and UI metadata are part of the interface.

Read AGENTS.md for the full spine contract, or browse docs/ for the vision, ADRs, and design plans behind it.
