All-Flow Model Evaluation Lanes

Extend the candidate-plus-evaluator loop from ci-diagnose to every agentic Bitterblossom workload that benefits from model comparison, without changing the Rust spine or multiplying production side effects.

Boundary. Base tasks keep their webhook, cron, and canonical gate roles. New candidate variants are manual-only lanes for measurement and model evaluation.

Target Cohorts

Flow Current default Add Why
build Codex / GPT-5.5 subscription Kimi K2.7 Code, GLM 5.2 Compare subscription builder against API-auth open-model builders.
review Kimi K2.6 minimal DeepSeek V4 Pro, GLM 5.2 Compare PR review quality across model families without posting duplicate comments.

correctness, security, simplification, product Existing storm member models Two manual variants per kind Evaluate reviewer quality without registering duplicate canonical verdict kinds.

Implementation

  • Add focused agent TOML files with role/skills copied from their base flow.
  • Add manual-only task directories with budgets and hosts distributed across sprites.
  • Reuse base task cards through symlinks so prompt fixes apply to the whole cohort.
  • Document all first-class cohorts under docs/model-evals/.
  • Strengthen Rust tests to assert at least three materially different configs per evaluated flow.

Stop Conditions

  • No duplicate verdict declarations; bb check must keep one canonical task per gate kind.
  • No new webhook or cron triggers on candidate variants.
  • No Rust semantic workflow layer; config, cards, docs, and tests only unless validation forces otherwise.
  • Do not claim a model winner without real candidate run receipts and a model-eval run.
