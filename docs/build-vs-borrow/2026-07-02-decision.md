# Build-vs-Borrow Decision: Workflow Runtime and Substrate

Date: 2026-07-02
Backlog: 058 child 5

## Verdict

Keep Bitterblossom's custom Rust spine and the Fly/Sprites substrate baseline.
Do not adopt Temporal, Inngest, Trigger.dev, or Cloudflare Workflows as the
runtime for the plane in this pass. Do not add a second substrate adapter from
this pass.

Borrow patterns, not dependencies:

- keep the `bb-coding-harness-probe.v1` evidence packet as the acceptance test
  for future substrate candidates;
- use durable-runtime docs as pressure to keep queue, retry, visibility, and
  recovery contracts explicit;
- consider Cloudflare Sandbox/Agents, E2B, Modal, or Daytona only after a live
  probe beats the current local/Sprites contract.

## Workflow Runtime Comparison

| Candidate | What it is good at | Fit against Bitterblossom | Decision |
|---|---|---|---|
| Temporal | Durable workflow execution, event history, activity retries, long-running orchestration, explicit timeouts. | Strong general workflow runtime, but adoption would add server/workers and replay semantics around a plane whose hardest boundary is side-effecting coding-harness execution on isolated workspaces. Temporal cannot remove the need for `bb`'s file-first task config, budget authority, substrate probes, host leases, or CLI/API contract. | Do not adopt. Borrow the discipline of explicit histories and timeout/retry boundaries. |
| Inngest | Durable functions, step-level retries, cron/webhook/event triggers, observability, flow control. | Excellent fit for app background jobs, but it would move queue/retry semantics into an external service while `bb` still owns ledger-before-ack, budget blocking, executing-phase recovery, and sandbox dispatch. The current Rust spine is already small enough that the migration cost is larger than the code it would replace. | Do not adopt. Revisit only if event volume or multi-tenant flow control outgrows the local ledger. |
| Trigger.dev | Long-running AI/background tasks, retries, queues, monitoring, elastic scaling, TypeScript task authoring. | Closest product-shape overlap for AI workflows, but it is a competing task runtime rather than a narrow primitive. Bitterblossom's task/agent/trigger source of truth is file config plus a Rust CLI; moving workloads into Trigger tasks would violate the current product boundary. | Do not adopt. Use as comparison pressure for run visibility and task ergonomics. |
| Cloudflare Workflows / Agents / durable fibers | Durable agent identity, local SQL, recoverable execution, Workflows for durable background processing, Sandbox integration. | The strongest future replatform candidate because Agents and Sandbox are explicitly agent-shaped. It still requires a Cloudflare runtime rewrite and does not by itself satisfy `bb`'s repo-owned workload config, local CLI, provider budget, and host/workspace lease contract. | Do not adopt now. Candidate for a future second-plane experiment only after a probe packet proves the coding-harness workload. |

## Why The Current Spine Still Wins

The hardened custom spine is not just a queue. It already owns the product
contracts that the workflow products do not remove:

- a durable run row before trigger ack;
- file-first tasks, agents, triggers, budgets, and policies;
- budget admission, in-flight spend policy, cap-breach kill/escalation;
- host/workspace leases tied to substrate identity;
- explicit pre-execute retry vs executing-phase no-auto-replay;
- recovery probes that classify `alive`, `dead`, and `unknown`;
- outbox notifications and stale progress visibility;
- stable `bb` CLI/API JSON for agents and operators.

The local probe added in child 4 proves the baseline failure behaviors without
external infrastructure:

- happy run succeeds with structured usage and `REPORT.json`;
- declared secret is not captured in logs or artifacts;
- zero-exit missing `REPORT.json` fails the run;
- timeout kills the executing harness after 60 seconds.

Replacing the spine would still require rebuilding those contracts around the
new runtime. The likely result is more surface area, not less.

## Migration Costs

Workflow runtime adoption would cost:

- new service/SaaS dependency and credentials;
- migration of run state, retry state, scheduling, and observability;
- duplicated budget/ledger authority unless `bb` remains the source of truth;
- a second operational UI unless every state projects back into `bb`;
- a harder public repo story if product config drifts into vendor dashboards.

Substrate adapter adoption would cost less because the adapter seam already
exists. The right migration shape, if one candidate later wins, is a second
adapter behind the existing `WorkspacePlan` contract, not a runtime rewrite.

## Outcome

Outcome selected from the 058 oracle:

> keep custom spine and Fly/Sprites baseline with explicit proof

No runtime dependency is added. No architecture-changing follow-up is emitted
from this pass. Future substrate exploration should be new backlog only if a
candidate probe packet beats the baseline on the scoring rubric.
