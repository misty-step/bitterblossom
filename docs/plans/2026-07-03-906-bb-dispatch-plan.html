Ship the first two ergonomics children for `bb` as the lead's dispatch surface.

Add one fire-and-exit command, bb dispatch --repo <path> --brief <file>, and one follow command, bb logs -f <run-id>. Keep the runtime generic: brief text becomes the run payload for an existing manual task, logs read the existing ledger/artifact truth, and no steering or completion hand-back ships in this slice.

Outcome

An operator can start a trivial real repo job with one command, see the run id immediately, and follow the transcript/events until a terminal state without opening the dashboard or spelunking artifact dirs.

Proof

CLI contract tests, full ./scripts/verify.sh, a local-plane dispatch/log transcript, and a deployed-plane dispatch/log transcript.

Stop Rules

Stop if this requires workload-specific Rust, secret values in payload/argv, weakening gates, or implementing child 3 steer / child 4 hand-back.

Chosen Shape

SurfaceImplementationWhy
bb dispatchThin CLI convenience over a manual task dispatch. It builds a structured payload from repo path, brief file contents, optional model, and label, then persists the run and starts dispatch through the existing local CLI path.One command for the target user, no new workload brain in Rust.
bb logs -fPoll the existing run bundle, print new run events and released text artifacts such as stdout.txt/stderr.txt, then exit at terminal state.Uses ledger/artifacts as truth; avoids a second streaming/event store.
DefaultsDefault task is the checked-in manual builder unless a plane config declares a more specific dispatch task; default label is derived from the brief filename.Matches current product direction while keeping flags small.

Acceptance

  • bb dispatch --repo <path> --brief <file> prints a run id and exits.
  • --model and --label are accepted and appear in the job payload.
  • bb logs -f <run-id> follows until terminal state and prints a useful transcript.
  • Local binary and deployed plane both exercise the new path.

Quality System

  • Live oracle: trivial real dispatch plus log follow transcript.
  • Structural gate: focused tests and ./scripts/verify.sh.
  • Critic: fresh-context diff review against CLI/workflow risks before merge.
  • Boundary: children 3 and 4 remain backlog, not hidden flags.

Risks

  • Dispatch defaults could accidentally encode Misty Step instance policy in product code.
  • Log follow could miss live output if artifacts only release after completion; tests must make that behavior honest.
  • Deployed-plane proof may expose missing instance config, which should be reported as an operational gap instead of hacked into source.
